Gitlab: Create a merge request to update the master branch

When a sufficient amount of changes have been pushed from local instances to the feature branch of the Redrock-Swagger repository—and the feature branch is stable—it is necessary to move those features onto the master branch. The process of moving information from one branch to another is a merge request.

Up until this point, most of the processes have been carried out from a command line. Creating a merge request can also be executed from the command line, but for simplicities sake, this article will describe how to create a merge request via the Gitlab UI.

Create a merge request

For consistency, do not attempt to make a merge request unless given specific authorization from a repository owner.

  1. Start by navigating to https://gitlab.workfront.tech/ and using your Adobe credentials to log in.
  1. From the main repository overview, select the merge request icon from the menu at the left-hand side of the screen.
  1. After clicking merge request from the main menu, you will arrive at the merge request dashboard. Using either of the New merge request buttons, navigate to the next screen.

  1. You will be prompted to specify a Source branch and a Target branch
    1. To update the master branch, select feature branch as your Source and the master branch is your Target.
    1. Once you have specified the Source and Target, submit the information and move on to the next page.
  1. The next page will prompt you to leave a description for your merge request. All of your commit comments will be included in the merge request, so this description should be about the merge request itself.
    • An example merge request description might say something like:
      • "Merging up to master. Primarily a collection of description updates added between 10/10/20 and 10/15/20" or
      • "Merging up to master. Updates to schema files to improve data layout"
    • Note: the merge request itself will have a time and date stamp and the person making the request will assign themselves as an owner. So it isn't a requirement to include these details.
  1. Scroll down to the next section and to the Asignees field, click the blue text that says "Assign to me". This will automatically populate the Asignees field with your Gitlab handle. All other fields can be left blank.

  1. All remaining fields on the form can be left at the default values. If you are satisfied with your merge request, scroll down and click the blue button labeled "Create merge request".
    • Note: creating the merge request doesn't initiate the actual merge request. Before the actual merge goes through an authorized user will have to manually start it.
  1. After the merge request is created, you will be taken to the actual merge request that you just created.
    • In this example, the merge request is a draft because there haven't been any actual updates between the feature and master branches, at the time this merge request was created.
    • If there had been changes ready, the green "Create file" button would instead contain the text "Initiate merge".
    • The data is only pushed from feature into master after the merge has been initiated.
  1. If you suspect that something is wrong after the merge went through, Gitlab allows you to revert the branches back to their original state. If you revert the branches:
    1. locate and fix the problems on the feature branch.
    1. perform a new merge request and in the description field, document any changes that were made to correct errors in between the reverision and the new merge request.
