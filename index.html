Core API Documentation Maintenance and

developer.workfront.com

represents a completion of several projects, spanning approximately 1.5 years. The documentation covers the primary tasks required to maintain the Core API documentation including: maintaining associated Gitlab repos, updating Open API spec and HTML files, and uploading files to AWS hosting where they are made available on developer.workfront.com.

I was the primary contributor for this documentation. All content and processes described in Core API Documentation Maintenance are my personal work or the result of direct collaboration with development teams across Adobe Workfront, except for two specific areas:

  1. The npm packages that are used to generate the OpenAPI spec files, which were created by an engineer named Trevon Redman.
  2. The pipelines that are used to push the Gitlab content to AWS hosting, which were created by Sr. Cloud Services Engineer Rob Weaver.

Note:The Core API Documentation Maintenance project was originally created in Notion. The version that exists here is an HTML export. The formatting is close to the original, but stripped of certain details, like particular icons, fonts, and spacing. Most of the files linked in the documentation are directly from this project. If there is a link to an outside project, that reference was not exported and will not exist or not open a link (for security reasons).

The AWS Toolkits for VSCode, JetBrains, and Visual Studio have been my main focus for the last 3 years. One of the unique challenges facing documentation for the Toolkits is that each Toolkit is essentially a gateway for many other AWS services. This can make launches complicated, but it has also given me the opportunity to work with many different AWS services and be a direct contributor on many high-profile projects.

Using the API to sync data for programs and services

Click here to view on Adobe Workfront One

Using the API to sync data for programs and services represents a collaboration between myself and the director of project management Darin Patterson (his title has changed since the Adobe Acquisition). Darin and I have worked peripherally together on a number of projects over the last couple of years, but this was something he requested specifically after meeting with customers during the 2020 Workfront convention.

The biggest challenge for this assignment was that Darin wanted the voice to achieve two main goals, which have some inherent conflicts:

  1. He wanted the voice to introduce Workfront Admins to event subscriptions (webhooks), and simultaneously prompt them to reach out to developers
  2. He wanted the documentation to be useful for developers that were interested in implementing new or improving their existing event subscription experience

In order to address these issues I:

  • used a conversational tone
  • included a brief introduction that describes what event subscriptions and their value, for the layman
  • included links to other articles which go into more technical depth about event subscriptions and the examples covered in the article
  • Included basic instructions and technical information that is are necessary for developers to implement basic event subscription processes

The general text in the article is my own and due to time constraints, I was primarily editing on the fly. The code examples included throughout the document were provided by Darin, but for fun I also tested some of them using my own instance of Workfront and Postman. Darin also provided the technical editing for the piece, making sure that the text achieved his goals.

Rich text fields in the Adobe Workfront API

Click here to view on Adobe Workfront One

Rich text fields in the Adobe Workfront API represents a collaboration with several development teams. Some of those teams are based out of the US but others are located in Armenia. I’ve actually worked pretty frequently with the Armenia based developers because over time they’ve become the unofficial stewards of the Adobe Workfront Core API.

Among the biggest challenges for this assignment was attempting to satisfy all of the developers, managers, and UX people that were involved with the project. Normally, projects that deal with the API aren’t so complicated (in terms of administration), but this feature was deemed to be good for marketing. So, after writing the initial article, which only took about 1 week, the editing process continued on for about another month, if I remember correctly.

While I feel that some phrases and styling aren’t representative of me as an individual writer, I wanted to include this piece as an example of my ability to collaborate with many different teams and people simultaneously.

Some examples of this:

  • I was opposed to including the image files as examples, but this was a decision that was required by marketing
  • In one of the image files containing bold text, there is some slightly awkward wording, like the phrase: “This is a regular text” or “This is my first rich text,” which were created by one of the development teams
  • I don’t like the notes about features with no release dates. However, these were no-compromise asks from the marketing team
  • I would have chosen to omit either the pretty or the unformatted JSON responses, to make the article feel less busy. This was, however, a specific ask by PMs in an effort to make the documentation forgiving for Workfront admins, in hopes that it would spark conversation between their developer resources.

Other than that, a bulk of the writing and general basic (grammar, spelling) edits are my own. Code examples and images were provided by dev teams.
